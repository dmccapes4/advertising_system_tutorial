The machine under the system

Exposure, not mastery. Every service does one move: register a port → a daemon listens → the kernel allocates a resource. The three differ only in what they ask for.

building manager allocates resources

2 · Storage: fast & small ↔ slow & big

VRAM
fastest, smallest, volatile — model weights while thinking (wide bus feeds 1000s of cores in parallel: matrix math)
RAM
fast, medium, volatile — running programs, query working sets
DISK
slow, huge, persistent — the database files, the books, your .env
A model lives on disk but must be loaded into fast memory to think. Big context = more memory on top of the weights. Spills to CPU → slow.

Apple Unified Memory vs Nvidia GPU

Two bets on where fast memory lives. Apple shares one big pool; Nvidia keeps a separate VRAM card — but that pool is often smaller per GPU.

VRAM is isolated. Postgres and Chrome don't eat GPU memory — but the model is capped at 24 GB (typical). Top-end cards (H100 80 GB) win on raw bandwidth; scale by adding cards.

CPU offloading — when the model doesn't fit

If weights + context exceed VRAM (or the free slice of unified memory), Ollama can spill layers to system RAM and run them on the CPU. It still works — slowly.

GPULayers 1–20 on VRAM / GPU coresfast — matrix math
CPULayers 21–32 offloaded to RAMslow — crawls

Rule: fits in fast memory → fast generation. Spills to CPU → tokens trickle out. On Apple, a bigger unified pool helps you avoid this; on Nvidia, you hit the VRAM wall sooner.

4 · HTTP vs. Stream

HTTP — ask once

Handshake, one response, hang up. Perfect for CRUD: POST create, GET read, PUT/PATCH update, DELETE.

Click to send one request — response in ~½ s, then hang up.

SSE — running commentary

One connection stays open; the server pushes events until done. For slow workflows.

Click to open the connection — events arrive every 1–2 s until end.
